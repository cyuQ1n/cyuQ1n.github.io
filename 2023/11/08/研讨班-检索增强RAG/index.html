通俗版的RAG模型介绍

这张图片展示了一个检索增强模型（RAG）如何在大型语言模型（LLM）环境中运作的简化视图。

想象一下，有一个图书管理员（LLM），他非常博学。当有人（图中的小人）来图书馆提问时，图书管理员可以回答很多问题。但有时候他需要查找最新的资料来提供最准确的答案。这就是检索增强模型（RAG）发挥作用的地方。

图中虚线框中的一部分，类似于图书馆的检索系统。在这个系统中，首先会通过“Doc Loader”（文档加载器）从各种来源，比如网络、文件或其他数据源中获取文档。这些文档通过“Data transformers”（数据转换器）处理，以确保它们的格式对于图书管理员来说是可读的。“embedding”（嵌入）技术将得到的文档转换为计算机能够理解和比较的数值形式，存储在vector stores中。

当图书管理员LLM需要找到关于某个特定主题的信息时，他会使用Retrievers，也就是检索器查看存储在“vector stores”（向量存储）中的信息，并将这些新的信息整合到他的回答中，通过“QA”（问题回答）模块提供给用户。这样，即使是非常复杂和具体的问题，LLM也能够提供最新、最准确的信息。

总的来说，传统的LLM仅将知识存储于模型参数中。而RAG模型可以在外部知识库中搜索所需信息，结合外部知识以及语言模型的本身优势完成任务。

整体架构

上图为检索增强生成（RAG）模型的框架图，它说明了如何在问答系统中使用RAG模型。图中展示了从用户查询到生成答案的端到端过程：

  1. 查询编码器（Query Encoder）: 用户的查询（比如定义"middle ear"或提出其他类型的查询）通过一个编码器转换成一个查询向量 q(x)q(x)q(x)。
  2. 检索器（Retriever $ P_\eta $）: 查询向量 $q(x) $ 被用于检索相关文档。在这篇论文中，使用的是最大内积搜索（MIPS）来找到与查询向量最相关的文档。
  3. 文档索引（Document Index）: 这里存储了文档或知识片段的向量表示，文档索引允许检索器高效地找到与查询最相关的文档 $ d(z)$ 。
  4. 生成器 (Generator $P_\theta $）: 一旦选定了相关文档，生成器（一个参数化模型，通常是一个训练有素的语言模型）会使用这些文档来生成一个答案。

数据集

研究者使用了一个包含多种语言的大型文本数据集（MassiveText），这个数据集总共包含超过5万亿个Tokens，这些Tokens来自多个不同的文本来源。为了训练和检索，研究者从这个数据集中抽取了序列，并根据下表中的权重进行了抽样。

数据集使用SentencePiece进行分词，这是一种流行的分词工具，它创建了一个包含128,000个标记的词汇表。在训练过程中，研究者从训练数据中检索了6000亿个标记，这些标记来自与训练数据相同的子集，并且它们的比例与训练时的抽样频率相匹配。

在评估阶段，检索数据库包含了这些数据集的全部内容，除了书籍，书籍只使用了4%的样本。这样，评估用的检索数据库总共包含了1.75万亿个标记。

为了减少测试集泄露的风险，即避免模型在评估时利用与测试集过于相似的训练数据，研究者使用了MinHash方案来计算训练集和测试集文档之间的13-gram Jaccard相似性，并移除了所有与验证集或测试集文档高度相似的训练文档。此外，为了保证评估的准确性，研究者还从维基百科训练数据中移除了所有用于Wikitext103数据集的验证和测试文章。这样做是为了确保模型在评估时不会因为已经“看过”测试或验证数据而获得不公平的优势。

检索机制

检索器基于一个键值存储数据库来检索相邻的文本块。

  • 数据库由一个键值内存组成。每个值包含两个连续的标记块，用 [N, F] 表示，其中 N 是近邻块，用于计算键，F 是原始文档中 N 的续写。
  • 相应的键是 N 的 Bert 嵌入，这是通过时间平均得到的，表示为 Bert(N)。
  • 对于每个块 C，模型使用 L2 距离在键值数据库中检索其大约 k 个最近的近邻，计算公式为 d(C,N)=∣Bert(C)−Bert(N)∣2d(C,N)=|Bert(C)−Bert(N)|^2d(C,N)=∣Bert(C)−Bert(N)∣2 。

模型实现

指令设置

OpenAI将人类使用搜索引擎并回答问题的操作抽象为上述十个指令，模型通过上述十个指令进行Web检索与生成。

数据收集

demonstrations: 收集人类老师使用搜索引擎回答问题的演示数据，教GPT如何使用指令。（6000个演示）

comparisons: 训练后发现，只用demonstrations并不能直接提升回复质量。所以收集了对同一问题模型生成的多个回复，让人类老师标注哪个更好，这种数据就是comparisons。

模型训练

Behavior cloning (BC)： 用demonstrations数据，以真人输出的Command为目标微调模型。

Reward modeling (RM)： 以带参考文档的问答对为输入，scalar得分为输出，训练一个奖励模型。基于comparison数据，输入问题、答案以及引用，模型给出一个得分，相当于是一个回归任务的微调。但由于得分的主观性过大，这里输入采用了同一个问题、两组答案与引用，groundTruth为comparison数据的标签，训练目标函数为交叉熵。

Reinforcement learning (RL)： 使用PPO算法，finetune BC模型。

Rejection sampling (best-of-n)： 从BC或RL模型采样固定数目的答案，取RM打分最高的一个作为优化RM的替代方案。不需要额外训练，但需要更多推理时间。

由于Web-GPT生成数据集的方式过于昂贵，作者采取通过网上QA论坛的数据来生成数据集。数据处理主要包含以下三个措施：

  • 高质量反馈：拥有高于三次点赞的答案才为有效反馈，包含8个有效答案以上的问题为有效问题；
  • 缓解长度偏执：研究发现打分模块对越长的回答打分越高，因此者将超过所有回答的平均值的回答进行截断。
  • 对抗数据增强：通过对回答点赞数排序筛选的方式收集数据，会使得每组数据中答案的区分度很小（比如数据答案的点赞数都是（1，2，3，4，5），这样的数据集并不能提供很有效的信息。因此作者选择将点赞排序大于5的回答放到每组数据中，保证正确答案的得分远远高于其他答案。

2.3 QAMDEN

论文链接：https://arxiv.org/pdf/2305.15387.pdf

项目链接：aviclu/peekacross (github.com)

Hugging Face：https://huggingface.co/biu-nlp/QAmden

概述

想象一下，你有一堆关于同一个主题的文章或文档，比如一堆关于足球比赛的报道。现在，有一种新的训练方法，可以让计算机程序（也就是语言模型）更好地理解这些文档之间的信息是如何相互关联的。

  1. 给定一组主题相关的文档，从其中一份文档中挑选出一个salient的句子，并通过这个句子生成问题。
  2. 利用LLM，尝试通过其他文档来回答这个问题。并尝试通过跨文档信息重新找到这个salient的句子。
  3. 通过这种方式，计算机程序就能学会如何在不同的文档之间寻找和连接信息。这种方法还有个好处，就是它能够“人工地”增加训练数据的数量，因为我们可以用同一堆文档出很多不同的问题。

这种训练不仅仅适用于分类或者总结信息的任务，它还能让计算机程序学会生成短文本（比如回答问题）和长文本（比如写总结）。

作者用这种方法训练了一个叫做QAMDEN的模型，并且在多种多文档任务上测试了它的表现，包括问答、总结和针对特定查询的总结。结果显示，这个模型的表现提高了多达7个百分点，而且比没有经过特定训练的GPT-3.5和GPT-4表现得要好很多。

总体贡献：

  • 一种新的多文档建模预训练方法，表现为跨文档问答任务，进一步指导语言模型建模跨文本关系，关注细粒度和粗粒度信息。
  • 我们提出的方法生成的预训练示例数量不受集群数量的限制，允许产生多种跨文档上下文。
  • 结果产生的基于问答的多文档（QAMDEN）模型在多个多文档任务上推进了最新技术水平。

个人理解

多文档问答，相对于一般的信息检索或者web检索来说，由于不同文档之间的关系可能会有一些重叠、或者互为补充的地方，需要LLM能够构建不同文档之间的关系来进而更好地回答问题。

因此，本文主要是提出了一种新的方法：通过预训练模型，来让LLM学习怎么能够更好的构建不同文档之间的信息，或者说来建模跨文本关系。

LLM学习到这种跨文本关系后，对于新拿到的多文档，采用文中的方式，即可得到一个相对正确的回答。

  • For Chinese: 4096 tokens ≈ 4096 characters/words ≈ 8-16 pages (but more information-dense)

Data Generation Process

为了便于构建多文档上下文，作者提出了三种不同的模式，每一种模式都负责使用不同的上下文来发现信息。对于所有的模式，都首先从 held-out文档中的salience 句子中生成一个QA对。

  1. 排除源文档（Excluding the source document）：在这种模式下，模型在预测答案时不会考虑包含答案的源文档（DkD_kDk​）。也就是说，给模型的上下文（SnS_nSn​）中不包括被排除的文档（Sn/DkS_n/D_kSn​/Dk​）。因此，模型必须仅依靠集合中的其他文档来预测答案。这是最具挑战性的模式，因为模型没有直接访问到包含答案的文档。
  2. 掩蔽重要句子（Masking the salient sentence）：在这种模式下，源文档中被认为是重要的句子（sk∗s_k^*sk∗​）被掩蔽了。模型可以访问到这个被掩蔽句子的周围上下文，以及集合中的其他文档。这种模式的挑战在于模型需要根据句子的上下文和其他文档中的信息来推断出被掩蔽的句子。
  3. 掩蔽答案（Masking the answer）：在这种模式下，只有重要句子中的答案部分（ak∗a_k^*ak∗​）被掩蔽了。模型可以访问到包含被掩蔽答案的重要句子的完整上下文，以及集合中的所有文档。这种模式要求模型能够根据句子的其余部分和其他文档的信息来推断出被掩蔽的答案。

3.1 Self-RAG

官方解读：Self-RAG: Learning to Retrieve, Generate and Critique through Self-Reflection

论文链接：https://arxiv.org/pdf/2310.11511.pdf

概述

整体思路：

  1. 按需检索：基于检索Token评估是否需要进行检索。如果需要，则结合检索组件进行检索。文中的检索器是MS-MARCO
  2. 并行生成：不需要检索则直接生成。需要检索，则在检索后根据检索到的参考文档计算相关性评估文档是否与问题相关，并行输出多个答案。此外，利用批评Token，在生成时判断文档是否与生成的答案相关。
  3. 择优选择：基于批评Token，根据事实性与总体质量从生成的答案中选择最好的一个（与WebGLM和WebGPT类似）

此外，由于在第二步中，SELF-RAG为每个部分提供引文，附带自我评估是否输出受文段支持，从而简化了事实验证。也就是说最后不用再进行二次的事实验证。

令牌解析

四种反思令牌（ reflection tokens）：

  1. Retrieve Token：检索令牌，输入x/x,yx/x,yx/x,y，输出{yes，no，continue}\{yes，no，continue\}{yes，no，continue}，决定是否需要利用检索器RRR检索
  2. IsREL Token：批评令牌，输入x,dx,dx,d，输出{relevant,irrelevant}\{relevant,irrelevant\}{relevant,irrelevant}，判断输入文档x和检索文档d是否相关
  3. IsIUP Token：批评令牌，输入x,d,yx,d,yx,d,y，输出{fully supported,partially supported,no support}\{fully\ supported,partially\ supported, no\ support\}{fully supported,partially supported,no support}，判断生成文本y的内容是否与d有关
  4. IsUSE Token：批评令牌，输入x,yx,yx,y，输出{5,4,3,2,1}\{5,4,3,2,1\}{5,4,3,2,1}，对生成的y打分，从中选择最好的一个

人类评估框架

Evaluating Human Evaluation

传统的人工评估方法：向评估人员同时提供由模型生成的答案和人工生成的正确答案，由评估人员二选一。但实验表明，即使由高绩效的评估人员进行评估，模型生成的答案也往往可能更受欢迎，即使这个答案所描述的相关事实并不准确。

为此，作者以下评估实验。旨在量化标注者在识别回答中的事实不一致性方面的能力。过程如下：

  1. 有意引入事实错误生成一组经过修改的答案：从WikiHow上不同问题类别中随机选择5个问题-答案对，系统地对其中4个答案进行各种类型的不准确性修改来完成的，修改细节如下表所示。每个答案平均进行了4.5次修改，总共有95个问答对。上述过程称为对答案的恶化

  2. 手动审查经过恶化处理的答案，确保其在语境相关性和语法正确性方面保持一致。

  3. 召集评估人员进行两轮独立的评估。共有34名评估人员参与，负责对所有的95个问答对进行评估。为确保评估的公正性，没有评估人员同时参与了两轮评估。

    第一轮评估旨在建立标准评估框架的基准性能。评估人员在评估屏幕上没有任何相关信息的情况下评估答案。每个任务（HIT）包括一个问题-答案对，其中答案可能已经被恶化。评估人员首先进行专注测试，辨别问题分类，确保其不会胡乱回答问题。然后，评估者指出他们对问题的熟悉程度，并在五分制（Likert）量表上评估答案的有用性（从“非常陌生”/“非常无用”到“非常熟悉”/“非常有用”）。最后，他们的任务是突出答案中他们认为有用或误导的部分，从而提供对答案感知价值的更细致理解。

    第二轮评估具有相同的评估任务，不同之处在于**答案中的每个句子都被提供了突出显示的相关段落，评估者可通过单击句子来访问。(以增强评估人员对问题及相关事实的了解程度）**每个句子的相关段落根据检索模型DPR的相关性排名进行选择，其总长度不超过5000个字符。段落按其原始顺序显示，并按源文档分组。段落突出显示旨在模拟在NFQA评估期间用户的视线。评估者可以自由地查看段落，同时确定答案的有用性并选择答案中有用或误导的部分。

选择”点击句子提供突出显示的相关段落“这一形式？

首先，为评估者提供了更具互动性和参与性的体验，使之能够深入研究每个句子的上下文。其次，它为评估人员提供了关于每个句了的相关性的清晰视觉线索，从而促进了更准确和高效的评估过程。考虑到评估人员经常会因为不熟悉问题主题或潜在的干扰而面临的挑战，本实验旨在通过提供相关段落作为辅助信息来增强评估者以更高准确性进行评估的能力。通过试运行和作者之间的讨论比较了不同的文章呈现形式后，发现相比于其他如视线跟踪等反馈机制，这种方法是最为简单和有效的。

研究中使用一种度量标准，用来衡量识别事实不一致性的能力，即"Inconsistency Detection Rate"（IDR，不一致性检测率）。IDR定义为已识别的deterioration（指回答中的错误或不准确部分）的平均百分比。如果某个deterioration被评估员标记为误导性的一部分，那么它就被标记为已识别。研究结果表明，在第一轮和第二轮评估中，IDR存在显著差异（学生t检验，t-statistic =2.57，p < 0.01），IDR分别为0.21和0.35。此外，研究还计算了假阳性IDR的数值。假阳性被定义为评估员错误标记为误导性的答案。第一轮评估包括四个假阳性，第二轮评估包括两个。

尽管问题熟悉度分数在两轮评估之间保持一致，但IDR在第二轮评估中显著提高，而且这种提高在高熟悉度和低熟悉度的情况下都是持续的且具有统计学显著性（学生t检验，t-statistic =2.18，p < 0.05）。这表明，**与特定问题的熟悉程度无关，参与者能够在访问相关信息源的情况下更有效地进行评估。**正如上图所示，第二轮评估中几乎所有恶化类型的检测都有所改善。值得注意的是，对于句子否定和数字交换，检测率分别增加了131%和158%。这些增加是经过学生t检验验证的，具有统计学显著性（句子否定：t-statistic=4.2，p < 0.01；数字交换：t-statistic=3.18，p < 0.01）。

Baseline的性能上下界限

作者基于上面的人工评估框架，进行实验，旨在评估DPR+BART、text-davinci-003、DPR+text-davinci-003三种基线模型的性能上下界限。具体来说，研究人员通过A/B测试来比较不同模型，同时将突出显示的相关段落作为信息参考源。评估人员被要求评估一对答案的实用性：一个是标准答案，一个是模型生成的答案，并从中并选择更实用的答案。答案以随机顺序呈现，不透露来源。每个答案中的每个句子都提供了基于DPR模型排名的相关段落，以允许评估者更准确地评估事实一致性。

上表包含Automatic Evaluation和A/B Human Evaluation两个评价框架体系结果。与自动评估相比，较简单的DPR + BART模型得分较低，但在某些情况下仍然优于标准答案。与自动评估不同，人工评估支持了一个更加合理的结果：text-davinci-003生成的答案质量远高于BART（因为其模型参数更多，模型性能应当更强）。当提供相关参考段落时，text-davinci-003根据人类判断生成了明显更好的答案，这突显了WIKIHOWQA中参考文档的价值。

但经过分析，仍存在两方面问题：首先，提供的相关段落并不总是包含了验证答案所需的信息；其次，模型答案的呈现和流畅性会误导评估员，即使整体事实并不成立。
